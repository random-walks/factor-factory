# Implementation plan

Phase-by-phase roadmap with explicit acceptance criteria. Each phase
is a verifiable milestone — when its acceptance criteria are met,
the next phase can start. Phases CAN overlap where sensible (e.g.,
docs-writing can happen alongside engine development).

The full design rationale is in [`00_design_rationale.md`](00_design_rationale.md).
The architecture this plan ships is in [`01_architecture.md`](01_architecture.md).

---

**Acceptance gate:** ✅ all docs in this folder exist + are

## Phase 1 — Skeleton + acceptance gate (target: 2-4 weeks)

### Deliverables

1. **Repo scaffolding**
   - `pyproject.toml` with default deps + optional extras stubbed
   - `factor_factory/` package layout matching
     [`01_architecture.md`](01_architecture.md)
   - GitHub Actions: `pytest` on push, `ruff` lint, `mypy` strict
   - `_fixtures/synthetic_panels.py` with at least 3 canonical
     panels (small treatment-effect, no effect, large effect with
     spillover) used by all engine tests

2. **Tidy layer**
   - `factor_factory.tidy.Panel` with full pydantic schema (see
     [`03_specs/panel_contract.md`](03_specs/panel_contract.md))
   - `Panel.from_records(records, *, geography, freq, treatment_events) -> Panel`
   - `factor_factory.tidy.geography.{load_boundaries, centroids_from_boundaries, distance_matrix}`
   - `factor_factory.tidy.record_view.RecordView` companion class
   - Validation invariants: every panel passes `validate()` on creation

3. **Diagnostics layer (the four we use everywhere)**
   - `standardized_mean_differences(panel, *, treated_col, covariates) -> pd.DataFrame`
   - `parallel_trends_plot(panel, *, treated_col, time_col, outcome) -> matplotlib.Figure`
   - `multi_index_assertions(panel)` — raises on shape violations
   - `residual_diagnostics(residuals) -> dict` (Jarque-Bera + Shapiro-Wilk + Breusch-Pagan in one call)

4. **First DiD engine (proof of concept)**
   - `factor_factory.engines.did.twfe.TwfeEngine` adapter
   - `factor_factory.engines.did.estimate(panel, methods=("twfe",), ...) -> list[DidResult]`
   - `DidResult.summary_table()` for side-by-side comparison
   - Conformance test in `tests/test_engines/test_did_conformance.py`

5. **First-class jellycell integration MVP**
   - `factor_factory.jellycell.cells.setup(*, also=None) -> dict` —
     workaround for jellycell #J1 (the cache-skip footgun)
   - `factor_factory.jellycell.figure.from_path(path, *, caption, name=None)` —
     path-only `jc.figure` substitute (jellycell #J2)
   - `factor_factory.jellycell.tearsheets.{methodology, diagnostics, findings, manuscript, audit}`
     — at least stub renderers that produce non-empty output
   - `python -m factor_factory scaffold <name>` — generates a working
     showcase skeleton with notebook stubs + jellycell.toml + manuscripts/

6. **Documentation**
   - `README.md` — already exists, may need refresh after Phase 1
   - `docs/getting-started.md` — scaffold a showcase, run it, view it
   - `docs/jellycell-integration.md` — the conventions + the workarounds
   - **Out of scope for Phase 1**: API ref docs, Sphinx site, anything
     beyond hand-written markdown

### Acceptance gate

The single dogfood test that gates moving to Phase 2:

> **Scaffold a new showcase via `python -m factor_factory scaffold`
> against factor-factory v0.1 such that:**
>
> - All cells in `notebooks/01_load.py` run cleanly under
>   `jellycell run`
> - `jellycell render` produces an HTML catalogue
> - `jellycell lint` passes
> - The notebook source is **shorter** than an equivalent hand-rolled
>   pipeline
> - The four manuscripts (METHODOLOGY, DIAGNOSTICS_CHECKLIST,
>   FINDINGS, MANUSCRIPT) are auto-generated by
>   `factor_factory.jellycell.tearsheets.*`
> - At least one `DidResult` is produced via
>   `factor_factory.engines.did.estimate(panel, methods=("twfe",))`

If that works, Phase 2 starts.

### Risks for Phase 1

- **API shape churn during engine development.** The Protocol +
  result-dataclass shapes will need to evolve as we add more
  engines. Mitigation: mark v0.x as "no SemVer guarantees;" don't
  commit to v1.0 until Phase 4.
- **jellycell integration depth.** Risk of building too much
  scaffolding before we know what's actually needed. Mitigation:
  ship the MVP (scaffold + tearsheets stubs + cells helper), see
  what hurts in Phase 3 dogfood, expand from there.
- **Synthetic-fixture realism.** If the test fixtures don't match
  realistic panel shapes, conformance tests give false confidence.
  Mitigation: cross-check fixtures against actual downstream
  showcase panels.

## Phase 2 — Engine fan-out (target: ~2-3 months, one family per release)

Per release, ~2 weeks each:

| Release | Adds | Notes |
|---|---|---|
| **v0.2.0** | DiD: Callaway-Sant'Anna (`differences`), Sun-Abraham, Borusyak-Jaravel-Spiess | The full multi-estimator DiD set. |
| **v0.3.0** | RDD: rdrobust adapter | Canonical local-poly. Document the manual chi-square density-continuity workaround for rddensity (which is unusable on pandas 2.x as of 2026). |
| **v0.4.0** | SCM: Abadie original + pysyncon adapter + augmented (Ben-Michael-Feller-Rothstein 2021) | Multiple engines for single-treated-unit analyses. |
| **v0.5.0** | Changepoint: ruptures (PELT, BinSeg) + bayesloop (Bayesian) | Frequentist + Bayesian alternatives. |
| **v0.6.0** | STL: statsmodels + sktime MSTL + prophet | Multi-seasonality + decomposition-as-forecasting alternatives. |
| **v0.7.0** | Panel regression: linearmodels + pyfixest | The fast Stata-fixest port for hetero-robust SEs. |
| **v0.8.0** | Inequality: theil + decompositions + Oaxaca-Blinder | Distributional analysis. |
| **v0.9.0** | Spatial: Moran's I + LISA + spatial-lag DiD (consolidates esda + libpysal + spreg usage) | One canonical interface to PySAL. |
| **v0.10.0** | Reporting bias + Hawkes (with the identification-constraint doc baked into reporting-bias EM) | Tail-end methods. |

Each engine release MUST:

- Ship adapter modules conforming to the Protocol pattern
- Include conformance tests against synthetic fixtures
- Include at least one worked example in `docs/`
- Document the citation(s) for the underlying method (e.g., Callaway
  & Sant'Anna 2021, Calonico-Cattaneo-Titiunik 2014)
- Add the engine family as an optional extra in `pyproject.toml`

### Acceptance gate

The full stats-method survey is covered by adapter modules in
`factor_factory.engines.*` — every method (DiD, RDD, SCM,
changepoint, STL, panel-reg, inequality, spatial, reporting-bias,
Hawkes) has at least one canonical engine; DiD has all four named
alternatives.

## Phase 3 — Showcase rewrite (target: 2-3 weeks, parallel with late Phase 2)

Once factor-factory hits v0.4 (DiD + RDD + SCM all available),
rewrite the three downstream showcases against it:

1. **Rat containerization showcase** — biggest win. Has the most
   stats; expect ~50% line-count reduction. Multi-engine DiD comes
   for free; RDD via rdrobust replaces hand-rolled local-poly work;
   spatial-lag DiD via the `engines.spatial` adapter.
2. **Resolution equity showcase** — second-biggest win. Tidy +
   Theil + Oaxaca-Blinder all from factor-factory; spatial Moran's I
   via the spatial engine.
3. **Subway accessibility showcase** — minor changes (verbatim
   mirror, so mostly tearsheet template adoption + scaffold-command
   regeneration).

Each rewrite produces a "before vs. after" PR in the consumer repo
so reviewers can see the line-count + clarity improvement.

### Acceptance gate

All three showcases:
- Pass `jellycell run`, `jellycell render`, `jellycell lint`
- Generate the same headline JSON artifacts as the v1 versions
- Have shorter notebook source (the dogfood signal that the framework
  is pulling its weight)
- Use auto-generated manuscripts where possible (METHODOLOGY,
  DIAGNOSTICS_CHECKLIST, FINDINGS — the MANUSCRIPT stays
  hand-authored per project)

## Phase 4 — Upstream consumer migration (target: ~2 months)

### `nyc311` adopts factor-factory

- Add `factor-factory>=0.5` as a dep
- Port `nyc311.examples.case_studies.rat_containerization` and
  `resolution_equity` to use factor-factory's tidy + engines
- Existing `nyc311.stats.*` functions stay as **thin
  backwards-compat wrappers** that delegate to factor-factory
  adapters with deprecation warnings
- Bump nyc311 minor version (e.g., `0.4.0` if currently `0.3.x`) to
  reflect the new dep

### `subway-access` adopts factor-factory

- Same shape: add dep, port one example folder fully, deprecate
  homegrown stats behind warnings, bump minor version

### Acceptance gate

Both packages have at least one example folder fully ported to
factor-factory. Deprecation warnings emit on the legacy paths but
nothing breaks. End-users (currently just Blaise) can choose to
migrate at their own pace.

---

## Phase 5 — Major-version bump in consumers (when ready)

- `nyc311 1.0` / `subway-access 1.0`: drop the homegrown stats
  implementations entirely; `factor-factory` becomes a hard dep
- Document the migration path in each package's CHANGELOG
- Remove deprecation shims

### Acceptance gate

Legacy code paths deleted; no functional regressions in either repo's
example suite; SemVer 1.0 commitment for both nyc311 + subway-access.

## Cross-phase notes

- **Versioning**: `0.x` during Phases 1-3 (frequent breaking changes
  allowed); commit to v1.0 SemVer guarantees only after Phase 4
  settles. Document this on the repo.
- **Backwards-compat**: in factor-factory itself, every breaking
  change in v0.x must include a CHANGELOG entry + a one-line
  migration note. Once v1.0 ships, follow strict SemVer.
- **Coordination with jellycell** (historical, all resolved):
  jellycell issues #10–#15 were filed alongside factor-factory's
  v0.1 design work and addressed in parallel — all closed by
  jellycell 1.3.5. factor-factory's `jellycell/cells.py` and
  `jellycell/figure.py` stay as stable public-API shims (see
  `docs/jellycell-integration.md` → "Coordinated upstream items").
